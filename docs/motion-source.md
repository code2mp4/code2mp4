# Motion Source

**Motion source is the source code of a video.** It is the central artifact in the Code2MP4 pipeline — the editable, reviewable, version-controllable file that sits between creative planning (script, storyboard) and deterministic output (MP4).

## What is motion source?

In Code2MP4, motion source is a **standalone HTML file** containing:

- **CSS** — layout, colors, typography (bound to motion system tokens)
- **HTML** — scene containers, text elements, icons, cards
- **GSAP timeline** — animation, timing, easing, scene transitions
- **HyperFrames data attributes** — `data-start`, `data-duration`, `data-composition-id`, `data-track-index`

This is the same format that HyperFrames uses for its compositions. But in Code2MP4, it's not the starting point — it's the *middle layer*, generated by agents from structured upstream artifacts.

## Why motion source matters

### 1. It's editable

Unlike a black-box MP4, you can open the motion source in any text editor and change:
- Text ("the headline is too long")
- Timing ("move the CTA 2 seconds earlier")
- Colors ("the accent doesn't match our brand")
- Motion ("that entrance is too aggressive")

The agent that generated it can also edit it — the same way it edits code.

### 2. It's reviewable

You can diff two versions of motion source and see exactly what changed:

```diff
- tl.from("#headline", { y: 60, opacity: 0, duration: 0.7, ease: "power3.out" }, 0.3);
+ tl.from("#headline", { y: 40, opacity: 0, duration: 0.5, ease: "power2.out" }, 0.1);
```

No guessing what changed between renders. Git tracks it.

### 3. It's deterministic

Same motion source → same MP4 output, every time. No nondeterministic seeds, no random AI generation. This means you can:
- Re-render with different quality presets (draft → standard → high)
- Generate platform variants from the same source (16:9 → 9:16 → 1:1)
- Run in CI/CD and get the same output every build

### 4. It's agent-modifiable

Coding agents already know how to edit HTML, CSS, and JavaScript. Motion source is written in these languages. Agents can:
- Read the source to understand what a video is doing
- Edit the source to fix issues flagged by quality review
- Generate variants by modifying timing and layout
- Reuse patterns from one video in another

## Where motion source fits

```
brief.json        WHY      (goal, audience, constraints)
    ↓
script.json       WHAT     (hook, segments, pacing, CTA)
    ↓
storyboard.json   HOW      (scenes, visuals, text, motion)
    ↓
scene.json        DETAIL   (per-scene agent spec)
    ↓
motion source     EXECUTE  (HTML/CSS/GSAP — the source code)
    ↓
render.config     CONTROL  (quality, fps, resolution, variants)
    ↓
MP4               OUTPUT   (deterministic video file)
```

Motion source is the only layer that is *code*. Everything above it is structured data (JSON). Everything below it is generated output (MP4, quality reports).

## Motion source conventions

### Structure

```html
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=1920,height=1080">
  <style>/* CSS here */</style>
  <script src="https://cdn.jsdelivr.net/npm/gsap@3/dist/gsap.min.js"></script>
</head>
<body>
<div id="stage" data-composition-id="my-video" data-start="0" data-duration="30" data-width="1920" data-height="1080">
  <!-- Scene containers -->
  <div class="scene" data-scene="1" style="opacity: 1">
    <div class="scene-content">
      <div class="headline" id="h1">Headline</div>
    </div>
  </div>
  <div class="scene" data-scene="2">
    <div class="scene-content">
      <div class="headline" id="h2">Scene Two</div>
    </div>
  </div>
</div>
<script>
  window.__timelines = window.__timelines || {};
  const tl = gsap.timeline({ paused: true });

  // Scene 1
  tl.from("#h1", { y: 60, opacity: 0, duration: 0.7, ease: "power3.out" }, 0.3);

  // Transition 1→2
  const s2 = document.querySelector('[data-scene="2"]');
  tl.to(s2, { opacity: 1, duration: 0.5, ease: "power2.inOut" }, 5.0);
  tl.to('[data-scene="1"]', { opacity: 0, duration: 0.5, ease: "power2.inOut" }, 5.0);

  window.__timelines["my-video"] = tl;
</script>
</body>
</html>
```

### Non-negotiables

- `data-composition-id` on `#stage` div, NOT on `<html>`
- `window.__timelines["comp-id"] = gsap.timeline({ paused: true })` pattern
- No `Math.random()`, `Date.now()`, or `repeat: -1`
- Fixed viewport meta (`width=1920,height=1080`)
- Body dimensions match data-width/data-height
- Scene transitions use crossfade, wipe, or shader — not exit animations

## Motion source and the agent workflow

For the coding agent, motion source is the *output* of scene composition and the *input* to rendering. The agent's job is:

1. Read upstream artifacts (brief, script, storyboard)
2. Compose motion source from scene fragments
3. Run quality checks against the source
4. Pass the source to the render engine

The agent should never skip these steps. Motion source is not a black box to be generated blindly — it's a structured artifact to be built, checked, and maintained.
